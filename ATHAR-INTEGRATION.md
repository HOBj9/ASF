# تكامل أثر (ATHAR) — جلب مواقع الحافلات بشكل حي

هذا المستند يشرح بشكل عملي **كيف نجلب مواقع الحافلات بشكل حي** من منصة أثر داخل هذا المشروع، وما هو الموجود حالياً وما المطلوب لإكمال التتبع اللحظي حسب `IMEI`.

---

## 1) الوضع الحالي في الكود

### المصادر الفعلية الموجودة

- خدمة أثر الأساسية: `src/lib/services/athar.service.ts`
- API لجلب العلامات من أثر: `src/app/api/athar/markers/route.ts`
- API لجلب المناطق: `src/app/api/athar/zones/route.ts`
- لوحة البلدية التي تعرض الخريطة: `src/components/dashboard/municipality-dashboard.tsx`

### ماذا يتم جلبه الآن؟

حالياً يتم استدعاء أمر أثر:
- `USER_GET_MARKERS`

ومن ثم يتم إرجاع بيانات مبسطة بهذا الشكل:

```json
{
  "id": "...",
  "lat": 33.51,
  "lng": 36.27,
  "name": "..."
}
```

> هذا مناسب لعرض نقاط/ماركرز على الخريطة، لكنه **ليس ربطاً مباشراً مؤكداً بمركبة حسب IMEI** في الطبقة الحالية.

---

## 2) كيف تعمل دورة الجلب الآن

1. الواجهة تستدعي `GET /api/athar/markers`.
2. الـ API يحدد `branchId` من الجلسة.
3. يتم إنشاء `AtharService.forBranch(branchId)` باستخدام `atharKey` الخاص بالفرع.
4. يتم تنفيذ `USER_GET_MARKERS` وإرجاع النتائج للواجهة.
5. الخريطة تعرض الماركرز.

> حالياً هذا الجلب يحصل عند تحميل لوحة البلدية، وليس تدفق حي كامل (stream) للمواقع.

---

## 3) التتبع الحي (Live) — الطريقة المعتمدة

لتحقيق عرض حي عملي ومستقر:

### خيار A (الأبسط والأسرع): Polling

- تنفيذ استدعاء دوري لـ `/api/athar/markers` كل 5–15 ثانية.
- تحديث الماركرز على الخريطة مباشرة.

مثال واجهة:

```ts
useEffect(() => {
  let active = true;

  const loadMarkers = async () => {
    const res = await fetch('/api/athar/markers');
    if (!res.ok || !active) return;
    const data = await res.json();
    setMarkers(data.markers || []);
  };

  loadMarkers();
  const timer = setInterval(loadMarkers, 10000); // كل 10 ثواني

  return () => {
    active = false;
    clearInterval(timer);
  };
}, []);
```

### خيار B: WebSocket/SSE

- الخادم يجلب من أثر دورياً.
- يبث المواقع فوراً للواجهة عبر WebSocket.
- مناسب إذا أردت حركة لحظية جداً وعدد مركبات كبير.

---

## 4) التتبع حسب IMEI (المطلوب للأعمال التشغيلية)

إذا المطلوب: "أعطني موقع المركبة X حسب IMEI" بشكل صريح، فالأفضل إضافة طبقة API مخصصة.

### المطلوب إضافته

1. Method في `AtharService` لجلب objects/locations تتضمن `imei`.
2. Endpoint مثل:
   - `GET /api/athar/vehicles/location?imei=...`
3. توحيد مخرجات endpoint بهذا الشكل:

```json
{
  "imei": "123456789012345",
  "lat": 33.5123,
  "lng": 36.2788,
  "speed": 22,
  "course": 145,
  "timestamp": "2026-02-03T12:20:00Z",
  "online": true
}
```

4. ربط الناتج مع `Vehicle` محلياً (المحفوظ عندنا فيه `imei`) لعرض اسم المركبة/السائق.

---

## 5) المتطلبات الأساسية لنجاح الجلب الحي

- كل فرع يجب أن يملك `atharKey` صالح.
- كل مركبة محلية يجب أن تملك `imei` صحيح ومفعل في أثر.
- التأكد من أن حساب أثر الخاص بالفرع يرى نفس الأجهزة المطلوبة.
- مراقبة اللوجات عند الفشل (موجودة بالفعل في `athar.service.ts`).

---

## 6) التحقق السريع

### تحقق من API

- افتح:
  - `GET /api/athar/markers`
- يجب أن ترى `markers` تحتوي `lat/lng`.

### تحقق من التحديث الحي

- فعّل polling في الواجهة (كل 10 ثواني).
- حرّك مركبة في أثر.
- تأكد أن موقع الماركر يتغير على الخريطة.

---

## 7) ملخص تنفيذي

- الموجود حالياً يدعم جلب مواقع من أثر عبر `USER_GET_MARKERS` وعرضها على الخريطة.
- لجعلها **حيّة فعلاً**: نحتاج تحديث دوري (Polling أو WebSocket).
- ولعرضها **بدقة حسب IMEI لكل مركبة**: نضيف endpoint مخصص يعتمد على IMEI ويربطه بمركبات النظام.
